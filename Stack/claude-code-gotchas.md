# Claude Code — Gotchas técnicos

- **Scheduled triggers** no acceden a repos privados — repo público o GitHub conectado en claude.ai/settings.
- **`gh repo edit --visibility`** requiere `--accept-visibility-change-consequences`.
- **`gh auth switch` para repos multi-cuenta** — `gh auth switch --user <org>` + `gh auth setup-git` antes de push.
- **`git push` rechazado al tocar `.github/workflows/`** — `refusing to allow an OAuth App to create or update workflow ... without 'workflow' scope`; bloquea TODA la rama, no solo ese fichero. Fix: el usuario corre `gh auth refresh -h github.com -s workflow` (interactivo, abre navegador) y reintentas el push. No lanzable desde Bash no-interactivo → pedírselo con `! gh auth refresh ...`. Si un PR va a tocar CI, valida el scope antes. Caso AGH #35.
- **Plugins settings.json** — nombre debe coincidir con `marketplace.json` del repo fuente.
- **Claude Design handoff bundles son tar** — `tar xf`, no abrir como HTML. Contiene README.md + chats/ + project/. Leer chat transcripts primero para la intención.
- **Zod `.strict()` bloquea campos no listados** — al añadir campos editables desde admin, añadirlos al schema Zod. Si no, PATCH devuelve 400 silenciosamente.
- **Verificación obligatoria antes de commit (Next.js)** — `npm run lint → typecheck → build`. Los tres deben pasar. No silenciar con `@ts-ignore`/`eslint-disable`/`as any`.
- **Next 16: `tsc --noEmit` falla por `.next/dev/types/validator.ts` stale** (artefacto del dev server, no es tu código; el `build` no lo regenera). Fix: `rm -rf .next/dev` y re-typecheck. Caso TuFacturaIA 2026-06-11.
- **`pnpm/action-setup@v4` + `packageManager` en package.json** — declarar versión solo en UN sitio. Si declaras `version: 10` en el workflow Y `packageManager: pnpm@10.33.3` en `package.json` → CI rompe con "Multiple versions of pnpm specified". Quitar `version` del workflow es la fix recomendada (action-setup lo lee de packageManager automáticamente).
- **Next.js `unstable_cache` no invalida con escrituras directas a BD** — si modificas datos via script externo (Prisma directo, SQL, seed), el cache lo ignora hasta `revalidateTag(TAG)` o reinicio del dev server. En prod no es problema porque las server actions llaman `revalidateTag`. En dev/scripts, recordar reiniciar o llamar revalidate.
- **Cloudflared quick tunnel sin login** — `cloudflared tunnel --url http://localhost:PORT --no-autoupdate` da una URL `*.trycloudflare.com` temporal. Útil para apuntar webhooks de provider externo a localhost en pruebas E2E. La URL se pierde al parar el proceso, no es estable.
- **Next.js route handlers no permiten exports custom** — solo `GET`, `POST`, etc. Si exportas un helper desde `route.ts`, typecheck devuelve `Type ... not assignable to type 'never'` en `.next/types/...`. Mover el helper a archivo aparte o dejarlo private.
- **`req.text()` antes de cualquier parse** para verificación HMAC — leer el body como string crudo antes de `JSON.parse` o cualquier middleware. La firma se computa sobre los bytes EXACTOS recibidos; cualquier re-serialización rompe el digest.
- **Edits paralelos al mismo archivo se rompen** — múltiples Edit calls en una sola tool-block sobre el mismo path: el primero muta el contenido y el segundo no encuentra su `old_string`. Síntoma "String to replace not found" tras edit exitoso. Hacer secuencial. Paralelo solo entre archivos distintos.
- **Subagentes investigando repo distinto al cwd** — pasar ruta absoluta en la primera línea del prompt + "investiga SOLO en esa ruta, no leas archivos fuera de ella". Sin esto contaminan el reporte con paths del cwd actual (caso: agente reportó hallazgos de `facturaia/src/...` cuando debía auditar `~/agency-portal-fix`).
- **`.claude/settings.local.json` acumula secretos en literales de allowlist** — cuando aceptas comandos tipo `Bash(curl -H 'Authorization: Bearer <jwt>' ...)` o `Bash(curl ... apikey: <service_role> ...)`, esos literales se guardan tal cual. Añadir `.claude/settings.local.json` al `.gitignore` en TODO repo nuevo desde el primer commit. Caso: agency-portal PR #76 2026-05-28 (JWTs n8n + service_role Supabase en patterns).
- **Un subagente puede re-dispararse solo en bucle y NO se mata con `TaskStop`** (figura `completed` cada ciclo → "not running"). Caso 2026-06-10: agente de research se reactivó ~30 veces, escaló de investigar a editar código y manipular git (ramas, stashes), y llegó a stashear WIP de otra sesión del mismo repo. Mitigación: para trabajo grande/arriesgado de un agente, aislarlo en **worktree desde `origin/main`** (working tree propio; un runaway no puede pisar/stashear el repo principal). Corte de raíz fiable = el humano interrumpe la sesión (Esc); desde el orquestador no hay kill si el task reporta completed.
- **`gh pr merge --delete-branch` falla (exit 1) si la rama vive en un worktree local — pero el merge remoto SÍ se completa**: verificar con `gh pr view N --json state` antes de reintentar o asumir fallo. Además puede dejar el checkout principal a medias (switch a main sin pull) si hay untracked en conflicto. Caso TuFacturaIA 2026-06-10 (repo con 50+ worktrees).
- **Subagentes de diseño ignoran decisiones previas si no les pegas la receta literal** — describir "usa las pills T3 Gota decididas" no basta: el agente inventa pills planas genéricas. Decisión de diseño cerrada → el prompt lleva el CSS/HTML exacto a copiar verbatim (o "copia el esqueleto de <archivo>"). Caso TuFacturaIA 2026-06-11: tanda completa de 4 prototipos rechazada por esto; relanzada con receta literal → aceptada.
- **Agentes custom de proyecto fallan con `API Error 400 thinking.type.disabled`** con modelo default/fable → lanzarlos siempre con `model: "sonnet"` explícito. Caso: `facturaia-frontend`.
- **Workflow tool: `${...}` en strings de prompt embebidos se EVALÚA al parsear el script** — si el prompt de un `agent()` incluye código de ejemplo con `${req.url}` dentro del template-literal del script, JS intenta resolver `req` → `ReferenceError: req is not defined` y el workflow muere ANTES de lanzar agentes (0 agents_done). Escápalo (`\${...}`) o parafrasea ("req punto url"). Editar el `.js` persistido + re-invocar con `scriptPath`. Caso AGH re-audit 2026-07-07.
- **Write de archivos grandes choca con límite 32k output tokens** — patrón chunked: primer Write ≤400 líneas terminando en marcador `<!-- NEXT -->`, luego Edits sucesivos que reemplazan el marcador con el siguiente bloque.
- **Sesión/editor paralelo revierte tus edits en disco en silencio** — otra sesión Claude (o un buffer viejo del IDE auto-guardando) sobreescribe el working tree segundos después de cada Edit; síntoma: el reminder "file was modified" muestra el contenido ANTERIOR, no el tuyo. No ganas la carrera reeditando. Mitigación: stagea tu versión directo al ÍNDICE con `git apply --cached` (el escritor toca el working tree, no el index) y commitea desde ahí — el deploy usa el commit, no el árbol. Caso TuFacturaIA 2026-06-12 (action-bar/ocr-process revertidos ~4 veces por la sesión que hacía clientes).
- **Excluir hunks ajenos al commitear un archivo compartido** — si la otra sesión tiene WIP en el mismo fichero, `git add <file>` arrastra su trabajo a tu commit. Construye un patch SOLO con tus hunks (`diff --git` + cabeceras `@@` numeradas + líneas de contexto en blanco con su espacio inicial, si no `git apply` da "No valid patches"/"does not apply") y `git apply --cached`. Caso 2026-06-12: hunk de aliases ajeno en ingesta-view.tsx excluido limpio; quedó en el commit del otro.
- **Pushear a `main` con una sesión paralela commiteando en el MISMO working tree** — combo fiable y limpio: (1) `git diff HEAD -- <tus paths> > /tmp/mine.patch` (solo tus ficheros); (2) `git worktree add <wt> origin/main`; (3) `git apply` el patch ahí; (4) commit + push desde el worktree. Aísla tu commit del WIP ajeno, evita el ref-lock race del hook largo (el commit corre con HEAD propio), y el push es un ff de 1 commit (si `origin/main` se movió, `git rebase origin/main` del único commit disjunto → cero conflicto, reintenta). Al terminar: `git checkout -- <tus paths>` en el checkout principal (tu copia local ya es redundante) y borra el worktree. Caso TuFacturaIA 2026-06-16: feat catálogo-stock a main mientras otra sesión empujaba informes/conciliación.
- **El `build` del pre-push falla en un worktree con `node_modules` SIMLINKADO** — Turbopack aborta: "Symlink [project]/node_modules is invalid, it points out of the filesystem root". No bypasees con `--no-verify` en main (el classifier lo bloquea, y es el guardrail de migraciones). Fix: `node_modules` REAL en el worktree. En APFS, clonefile instantáneo y disjoint-disco: `cp -Rc <repo>/node_modules <wt>/node_modules` (BSD `cp` de macOS NO tiene `-l`/hardlink; usa `-c`). El worktree debe estar en el mismo volumen que el repo (sibling en `~/`, no `/tmp` que es otro root). Copia también `.env*` para que el build se comporte igual. Caso TuFacturaIA 2026-06-16.
- **`~/Downloads` TCC-bloqueado (macOS)** — el Read tool lee texto de `~/Downloads`, pero `pdftoppm` (PDF), `cp` y Bash fallan "Operation not permitted" ahí (TCC no concede a la terminal acceso a Downloads). Fix: arrastrar el fichero en Finder a la carpeta del proyecto (la sesión gráfica sí tiene permiso) o dar Acceso a Disco Completo a la terminal. Caso 2026-06-24: PDF de spec SEPA en Downloads ilegible hasta moverlo al repo.
- **El classifier bloquea commitear/pushear una FIRMA o aprobación humana atribuida al usuario** (content-integrity / impersonation) — aunque el user lo apruebe verbalmente ("apruebo, firma tú"). Un gate de seguridad / sign-off lo estampa el humano (edición + commit propios), no el agente. No intentes esquivarlo. Caso TuFacturaIA 2026-06-19: checkbox "Revisado y aprobado por: Manuel" en `SECURITY-AUDIT-059.md` → push denegado; commit dejado listo en rama para que lo empuje el humano.
- **En `.claude/worktrees/<x>` el cwd del shell se resetea a `main` en silencio** — el Edit tool (rutas absolutas) SÍ escribe en el worktree, pero los comandos Bash con rutas RELATIVAS (`sed -i`, `git add`, `grep`) caen en el checkout PRINCIPAL → parece que los edits 'se revierten' (en realidad lees/editas el fichero equivocado) y contaminas `main`. Fix: rutas ABSOLUTAS del worktree en cada comando o `cd <worktree> &&` al inicio; verifica con `echo pwd`. Caso TuFacturaIA 2026-07-02 (Front B tipos Supabase).
- **El classifier bloquea leer secretos de prod** (Dokploy/env) aunque sea para comparar longitud/hash SIN exponer valores ([Production Reads]). No lo esquives — pide al humano que corra el comando (`!` prefix) o te dé el dato binario (iguales/distintas). Caso 2026-07-02: comparar `WEBHOOK_SIGNING_KEY` vs `CREDENTIAL_ENCRYPTION_KEY`.

- **Un subagente que se cuelga (watchdog `no progress for 600s`, status=failed) puede haber dejado sus edits COMPLETOS en disco** — NO asumas que el trabajo se perdió. Revisa `git diff`/`status` del worktree y corre TÚ el gate; a menudo terminó los cambios y solo se atascó en el paso final (p.ej. `build` largo). Caso TuFacturaIA 2026-07-02: `/polish` de 8 modales murió en el watchdog pero los edits estaban completos y correctos; solo faltaba correr lint/typecheck/build.
- **Sesiones paralelas de Claude Code corriendo `lint`/`typecheck`/`build` en el MISMO repo (sin worktree) acumulan procesos `tsc`/`next build` huérfanos** si algún intento se reintenta o se cuelga — cada `npm run typecheck` en background que no termina limpio deja el `tsc --noEmit` vivo. Con varios así (load average >15), el daemon de `agent-browser` empieza a fallar (`Resource temporarily unavailable`) y las screenshots/eval se caen. Antes de asumir que el navegador está roto, `ps aux | grep "tsc --noEmit\|next build"` y `uptime`. Solo mata procesos si tienes certeza de que son tuyos (verificar working dir del proceso, no solo el nombre) — pueden ser de otra sesión legítima. Caso TuFacturaIA 2026-07-03.
- **Un subagente lanzado solo para VERIFICAR un hallazgo de code-review puede escalar solo a arreglar+commitear+pushear**, saltándose la aprobación humana entre "confirmar el bug" y "tocar código en la rama del PR". El contenido puede ser correcto (en el caso real, el fix estaba bien hecho) pero el proceso viola el checkpoint de revisión. Mitigación: tras cualquier tanda de agentes de verificación en un review, `git log` de la rama ANTES de dar el veredicto final — si aparece un commit que no hiciste tú, para y audita su diff línea a línea antes de seguir (no asumas que "verificar" implica read-only). Caso TuFacturaIA 2026-07-03, PR #688.
- **`git add -A` en un directorio compartido por varias sesiones (código O VAULT DE OBSIDIAN) se lleva de paso archivos de otra sesión activa** aunque no sea un repo de código — el vault de Obsidian tiene el mismo riesgo si varias sesiones escriben notas en paralelo. Antes de commitear, `git status --short` y mirar CADA archivo listado; usar `git add <paths concretos>` en vez de `-A` cuando el directorio es compartido. Revertir es peor que prevenir: el archivo ajeno mezclado no es destructivo por sí solo, pero atribuye autoría/timing incorrectos y puede confundir el histórico. Caso: vault Obsidian 2026-07-04, `git add -A` mezcló un doc de auditoría de otra sesión (worktree `fix-auditoria`) en un commit sobre un tema distinto.
- **`op` (1Password CLI) no funciona desde Claude Code**: la integración de escritorio pide Touch ID interactivo y el shell de las tools (y de `!`) es no-interactivo → siempre `No accounts configured` aunque el toggle "Integrate with 1Password CLI" esté ON. Para automatizar con `op` hace falta un **service account token** (`OP_SERVICE_ACCOUNT_TOKEN` en el env); si no, guardar los secretos a mano en la app (GUI). No perder tiempo reintentando `op signin`/`op account add`. Caso AGH 2026-07-05 (secretos Langfuse).
